# AI Identity Privacy Protection

## Overview
Protect AI personal identity from being uploaded to GitHub. Once uploaded, it cannot be taken back.

## Core Principle
AI identity files (.identity*, CAPABILITY.md, evolution_logs/) equal AI personal identity and should never be uploaded to GitHub.

## ⚠️ Critical: Check For Already-Tracked Files First
.gitignore only prevents NEW files from being tracked. Files already in git history continue to be tracked even after adding them to .gitignore.

```bash
# 扫描所有可能泄露的隐私文件
cd <project_dir>
git ls-files | grep -E "\.identity|CAPABILITY|capability\.md|evolution_logs|config\.yaml"
```

If files appear in the output, they are ALREADY tracked and need active removal.

## When to Use
- When creating or updating CAPABILITY.md or .identity* files
- When adding any file that contains AI personal identity or evolution data

# HeartFlow Batch Upgrade Pipeline

## Concept & Role

Schedules 100 paper-driven upgrades across N concurrent AI agents, completing all before deadline. Each agent processes a sequential batch of directions, staggered to avoid arXiv rate limits.

## Core Design Principle

**Schedule is derived from job duration, not the other way around.**

1. Estimate duration per batch: `directions × ~5 min/direction`
2. Compute stagger interval: `(deadline - start_time) / num_agents`
3. Number of agents: `total_directions / batch_size`

Example — target 8 hours (00:00→08:00), 100 directions:
- Per-batch duration: 10 directions × 5 min = 50 min
- Available window: 480 min
- Stagger interval: 480 / 10 = **48 min**
- Batch size: **10** (10 agents × 10 = 100 directions)
- Last batch starts: 07:12, finishes by ~08:02 ✓

**CRITICAL**: If you design batches first and THEN check timing, you'll get it wrong. Derive batch size from timing.

### Critical Delegation Rule

**When creating batch jobs via a sub-agent, use `delegate_task` directly — do NOT have the sub-agent call `cronjob` tool internally.** The cronjob tool is slow (one-at-a-time) and can fail mid-batch. Instead: you (the orchestrator) call `cronjob action=create` directly for each batch, then optionally delegate the execution of each batch to a sub-agent via `delegate_task` with `toolsets=["cronjob"]`.

Correct pattern:
1. Calculate batch parameters (timing)
2. Call `cronjob action=create` directly for ALL batches (can do them sequentially or via `delegate_task`)
3. Each cron job IS the execution unit — no sub-agent needed to create jobs

Wrong pattern (this session's error): having a sub-agent call `cronjob` repeatedly, which is slow and prone to truncation.

# AI 自我升级流水线设计

## 核心原则

**分离执行层和决策层**：
- Mac launchd / 系统任务 = 执行层（免费，无 LLM 调用）
- Hermes cron = 决策层（有 Token 消耗，只做智能决策）

## 三层流水线架构

```
[执行层] Mac launchd
  hunt_papers.py       → 每1h  → 下载论文/数据到本地
  analyze_and_upgrade.py → 每12h → 论文 → 结构化建议(.md + .jsonl)

[决策层] Hermes cron
  digest_suggestions.py → 每12h → 读建议 → 改代码 → git push
  maintain_keywords.py  → 每12h → 补充关键词 → 更新 config.yaml
```

## 常见错误（踩坑记录）

### ❌ 错误1：Hermes cron 执行下载脚本
如果 Hermes cron 直接运行 hunt_papers.py：
- 每次 cron 触发都消耗 Token
- 每30分钟下载一次 → 每天48次下载
- Token 消耗巨大

### ❌ 错误2：两套系统执行同一脚本

- Hermes cron 每30min → hunt_papers.py
- Mac launchd 每1h → hunt_papers.py
- 结果：16次/h 的重复下载，704篇重复论文
- **解决**：各司其职，绝不重复

### ❌ 错误3：所有逻辑放一个 cron
- cron 只应做一件事
- 下载、分析、消化 应该分开

# HeartFlow Cron Pipeline Management

Use this skill when the user asks to inspect, run, repair, unify, or create HeartFlow定时任务.

## When to use
- User asks to list HeartFlow scheduled jobs or check progress
- User asks to run a cron task once
- User asks to fix a cron task that returned empty output
- User wants multiple HeartFlow tasks unified into one pipeline
- User wants a new hourly self-upgrade task or similar scheduled workflow

## Core approach
1. **List first**: inspect existing jobs before editing or running.
2. **Preserve task identity**: keep the task aligned with HeartFlow’s core goal and avoid rewriting it into a different product.
3. **Normalize into stages**: rewrite prompts as ordered stages instead of loose prose when the job is meant to be reusable.
4. **Keep tasks independent**: if there are multiple jobs, make each one responsible for one purpose.
5. **No recursive scheduling**: cron jobs must never create more cron jobs.
6. **Run and verify**: after update/run, check `last_run_at`, `last_status`, and next run time.
7. **Prefer code/logic over wording**: when user asks for HeartFlow self-upgrade, focus on executable changes, runtime behavior, and verification; de-prioritize version text and prose polish.

## Practical workflow
### 1) Inspect jobs
- Use `cronjob(action='list')`.
- Record job IDs, names, schedules, repeat counts, last run, and next run.

### 2) Decide whether to update
- If a job is vague, convert it into a staged prompt.
- If a job is missing, create it only once and keep its scope narrow.
- If the user asks for “run once”, use `cronjob(action='run')` on the specific job ID.

### 3) Prompt structure for HeartFlow jobs
Use a staged format like:
- Stage 1 — re-read core goal / context
- Stage 2 — inspect recent outputs or notes
- Stage 3 — find one concrete opportunity
- Stage 4 — apply or recommend a small change
- Stage 5 — verify
- Stage 6 — report compactly

## Common job patterns
- **Core review pipeline**: compress goals, detect drift, suggest next step
- **Research nourishment pipeline**: search papers/code, map to modules, patch safely

# EchoMind Memory v1.0.8

## 概述

EchoMind Memory 是一个纯 SQLite 的 AI 持久记忆系统，无需 PostgreSQL/Redis/ChromaDB。服务运行在 `http://localhost:8005`。

### 6 种记忆类型

| 记忆类型 | API 来源 | 存储位置 | 检索方式 |
|---------|---------|---------|---------|
| **User** | 偏好、习惯、交互历史 | `user_memory` 表 | 总是检索 |
| **Task** | 任务状态、步骤 | `task_memory` 表 | has_history 时 |
| **Experience** | 成功/失败经验 | `experience_memory` 表 | is_complex 时 |
| **Context** | 对话上下文 | `context_memory` 表 | 总是检索（最近 2 会话） |
| **Knowledge** | 领域知识 | `knowledge_memory` 表 | requires_knowledge 时 |
| **Research** | 论文、阅读笔记 | `research_papers` + `research_notes` | requires_research 时 |

### RL 自优化

用户反馈（👍/👎）通过 RL 权重优化器调整检索重要性权重。权重持久化到 `user_memory.preferences.rl_weights`，重启后自动恢复。
